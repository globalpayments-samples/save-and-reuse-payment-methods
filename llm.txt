# Save and Reuse Payment Methods

> Multi-language demonstration of payment method storage and one-click payment processing using Global Payments SDK. Shows how to convert single-use tokens to multi-use stored tokens with integrated customer data across PHP, Java, and .NET implementations.

This repository provides **working reference implementations** for:

- **Multi-Use Token Creation** - Critical pattern: `verify().withRequestMultiUseToken(true)` for card-save flows (NOT a minimal charge)
- **Payment Method Storage** - JSON-based persistence with customer data integration  
- **One-Click Payments** - Process charges using stored multi-use tokens
- **Mock Mode Support** - Test without credentials using simulated API responses

**⚠️ CANONICAL IMPLEMENTATIONS**: `PaymentUtils` files contain working examples that have been tested and validated. Modify with caution and test across all three languages when making changes.

**Target Audience**: Developers integrating Global Payments GP API for stored payment methods

**Tech Stack**: PHP 7.4+ | Java 11+ | .NET 9.0+ with Global Payments SDK (v13.1+ / v14.2.20+ / v9.0.16+)

**Key Pattern to Understand**: For card-save flows, use **`Verify`** (not a minimal charge) with `.withRequestMultiUseToken(true)` to create multi-use tokens. `Verify` performs a card brand-approved $0 auth. Using a small charge purely for tokenization is treated as misuse of authorizations by card brands and incurs interchange penalties. All three implementations demonstrate this pattern.

## Code Patterns Reference

Below are the complete working implementations of the critical patterns. These are extracted from the actual working code:

### Multi-Use Token Creation Pattern (PHP)

```php
// From php/PaymentUtils.php lines 83-130
public static function createMultiUseTokenWithCustomer(string $paymentToken, array $customerData, array $cardDetails): array
{
    // Create tokenized card from single-use token
    $tokenizedCard = new CreditCardData();
    $tokenizedCard->token = $paymentToken;
    $tokenizedCard->cardHolderName = trim($customerData['first_name'] ?? '') . ' ' . trim($customerData['last_name'] ?? '');

    // Create address from customer data
    $address = new Address();
    $address->streetAddress1 = trim($customerData['street_address'] ?? '');
    $address->city = trim($customerData['city'] ?? '');
    $address->province = trim($customerData['state'] ?? '');
    $address->postalCode = self::sanitizePostalCode($customerData['billing_zip'] ?? '');
    $address->country = trim($customerData['country'] ?? '');

    // CRITICAL: Use Verify (not a minimal charge) with withRequestMultiUseToken flag
    // A minimal charge for tokenization is misuse of authorizations per card brand rules
    $response = $tokenizedCard->verify()
        ->withCurrency('USD')
        ->withRequestMultiUseToken(true)  // This flag is REQUIRED for multi-use tokens
        ->withAddress($address)
        ->execute();

    if ($response->responseCode === 'SUCCESS' &&
        $response->responseMessage === 'VERIFIED') {
        
        $multiUseToken = $response->token ?? $paymentToken;
        
        return [
            'multiUseToken' => $multiUseToken,
            'brand' => self::determineCardBrandFromType($cardDetails['cardType'] ?? ''),
            'last4' => $cardDetails['cardLast4'] ?? '',
            'expiryMonth' => $cardDetails['expiryMonth'] ?? '',
            'expiryYear' => $cardDetails['expiryYear'] ?? '',
            'customerData' => $customerData
        ];
    }
}
```

### Multi-Use Token Creation Pattern (Java)

```java
// From java/src/main/java/com/globalpayments/example/PaymentUtils.java lines 247-285
public static MultiUseTokenResult createMultiUseTokenWithCustomer(String paymentToken, CustomerData customerData, CardDetails cardDetails) throws Exception {
    CreditCardData card = new CreditCardData();
    card.setToken(paymentToken);
    card.setCardHolderName((customerData.firstName + " " + customerData.lastName).trim());

    Address address = new Address();
    address.setStreetAddress1(customerData.streetAddress.trim());
    address.setCity(customerData.city.trim());
    address.setState(customerData.state.trim());
    address.setPostalCode(sanitizePostalCode(customerData.billingZip));
    address.setCountry(customerData.country.trim());

    // CRITICAL: Use Verify (not a minimal charge) with withRequestMultiUseToken flag
    // A minimal charge for tokenization is misuse of authorizations per card brand rules
    Transaction response = card.verify()
            .withCurrency("USD")
            .withRequestMultiUseToken(true)  // This flag is REQUIRED for multi-use tokens
            .withAddress(address)
            .execute();

    if ("SUCCESS".equals(response.getResponseCode()) &&
        "VERIFIED".equals(response.getResponseMessage())) {
        
        String finalToken = response.getToken() != null ? response.getToken() : paymentToken;

        return new MultiUseTokenResult(
            finalToken,
            determineCardBrandFromType(cardDetails.cardType),
            cardDetails.cardLast4,
            cardDetails.expiryMonth,
            cardDetails.expiryYear,
            customerData
        );
    }
}
```

### Multi-Use Token Creation Pattern (.NET)

```csharp
// From dotnet/PaymentUtils.cs lines 48-95
public static async Task<MultiUseTokenResult> CreateMultiUseTokenWithCustomerAsync(string paymentToken, CustomerData customerData, CardDetails cardDetails)
{
    return await Task.Run(() =>
    {
        var card = new CreditCardData
        {
            Token = paymentToken,
            CardHolderName = $"{customerData.FirstName?.Trim() ?? ""} {customerData.LastName?.Trim() ?? ""}".Trim()
        };

        var address = new Address
        {
            StreetAddress1 = customerData.StreetAddress?.Trim() ?? "",
            City = customerData.City?.Trim() ?? "",
            Province = customerData.State?.Trim() ?? "",
            PostalCode = SanitizePostalCode(customerData.BillingZip),
            Country = customerData.Country?.Trim() ?? ""
        };

        // CRITICAL: Use Verify (not a minimal charge) with WithRequestMultiUseToken flag
        // A minimal charge for tokenization is misuse of authorizations per card brand rules
        var response = card.Verify()
            .WithCurrency("USD")
            .WithRequestMultiUseToken(true)  // This flag is REQUIRED for multi-use tokens
            .WithAddress(address)
            .Execute();

        if (response.ResponseCode == "SUCCESS" &&
            response.ResponseMessage == "VERIFIED")
        {
            var multiUseToken = response.Token ?? paymentToken;

            return new MultiUseTokenResult
            {
                MultiUseToken = multiUseToken,
                Brand = DetermineCardBrandFromType(cardDetails.CardType ?? ""),
                Last4 = cardDetails.CardLast4 ?? "",
                ExpiryMonth = cardDetails.ExpiryMonth ?? "",
                ExpiryYear = cardDetails.ExpiryYear ?? "",
                CustomerData = customerData
            };
        }
    });
}
```

## Critical Implementation Notes

### Why Verify Instead of a Minimal Charge?

For card-save flows (where no immediate payment is being taken), the correct approach is `Verify` with `.withRequestMultiUseToken(true)`. `Verify` performs a card brand-approved $0 authorization that confirms the card is valid and binds multi-use permission to the token. Using a minimal charge (e.g. $0.01) purely for tokenization is treated as misuse of authorizations by card brands and results in an assessed interchange fee penalty to the merchant.

If the card-save happens as part of an initial purchase, a `Charge` with `.withRequestMultiUseToken(true)` can be used instead — that captures the payment and returns the multi-use token in a single call.

### Expiration Date Handling

All implementations handle both 2-digit and 4-digit expiration years. The SDK requires 2-digit format, so 4-digit years (e.g., "2028") are automatically converted to 2-digit (e.g., "28").

### Mock Mode

When no GP API credentials are provided, mock mode automatically enables. Mock responses in `MockResponses` files must match real API response structures exactly.

### Security Warnings

This is demonstration code with simplified storage:
- JSON file storage is NOT production-ready
- No encryption at rest for stored tokens
- No authentication/authorization on endpoints
- Single-tenant design without user isolation

For production, implement proper database storage, authentication, encryption, and PCI DSS compliance.

## Architecture

### Frontend Flow
1. User enters card data in Global Payments hosted fields
2. SDK generates single-use token (client-side, PCI-compliant)
3. Single-use token + customer data sent to backend
4. Backend converts to multi-use token and stores

### Backend Flow  
1. Receive single-use token and customer data
2. Create `CreditCardData` object with token
3. Create `Address` object with customer billing info
4. Execute `Verify` with `.withRequestMultiUseToken(true)` flag (card-save flow)
5. Store returned multi-use token with customer data
6. Return payment method ID to frontend

### Payment Flow
1. Frontend requests charge with payment method ID
2. Backend retrieves stored multi-use token
3. Execute charge using stored token
4. Return transaction result
